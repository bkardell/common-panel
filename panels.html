This document describes a universal markup attempting to pave the cowpath we observe for a construct called a <common-panel> which contains what appear to be common attributes and provide also and a single selection interaction paradigm for a collection of them called a <common-panel-set>. Defining these allow editors to create a reusable set of declarative code which remains semantic, widely stylable in visual terms while retaining a conceptual and physical separation that retains the important accessibility benefits for all of them.

Introduction

When editing a document, an author attempts to mark it up semantically and control a separation of visual aspects through CSS. This allows various the user-agent and various tools to use the same semantic data to provide any number of affordances for accessibility, navigation and interaction. Over time, UIs have created a number of metaphors for visually managing and providing single selection/focus navigation through a collection of titled 'blocks'.

Because the art of creating a good user interface is something we are all still trying to figure out, the history and development of these metaphors appear to have obscured the underlying fact that in virtually every way, a number of them are conceptually and in an very real sense practically nearly identical in every way (in fact, the WAI-ARIA Best Practices document defines accordions as a kind of tablist and in terms of keyboard navigation says that UAs should treat up/down and left/right navigation for tabs and accordions as interchangeable, for example) but are visually very different. With the advent of ever more variant screen sizes, interaction methods, responsive design, and a newly energized focus on accessibility, this question has begun to come to the forefront of the mainstream: We have designs which switch their visual representation based on the current size of the screen, rather than being constant. For example, on a wide screen, a tabbed interface may seem like a natural choice, whereas the same interface is more natural as an accordion, or a deck on a small one. This proposal attempts to uncover the underlying structures and interactions that accurately describe the common "thing" and provide ways and examples of achieving robust separation in which CSS plays its intended role and it is easy for authors to "do the right thing".

As an author, one would like to define a simple panel which has a title and have it display in a common fashion and have common UA/AT features.

To this end, we introduce a common markup element... 

<common-panel title="Panels Rock">
    <p>Don't you really think?</p>
</common-panel>

which exposes the following pseudo-elements with built in semantics and functionality, and which can be used to achieve a wide number of visual presentation effects:

  • ::--common-panel-header
  • ::--common-panel-icon
  • ::--common-panel-title
  • ::--common-panel-destroy (if the is-removable attribute is set)

<common-panel common-title="Removable Panels" is-removable>
    <p>Don't you really think they do? You can click the button in the titlebar to remove this panel.</p>
</common-panel>
        
<common-panel title="Collapsable Panels" collapse-state="collapsed">
    <p>Don't you really think they do? This element was collapsed initially, if you can read this, it's been expanded</p>
</common-panel>

When placed as a child of a <common-panel-set> element, the panelset will manage the selection/activation of a <common-panel> at a time. The collapse-state of a panel indicates whether it is intended to be selected or not (any value other than expanded is considered collapsed. By default, the first <common-panel-set> is considered expanded. Because the parser processes elements in document order, any subsequent claims of expansion trump the first by removing the attribute from siblings. 

<common-panel-set id="foo">
    <common-panel title="Panel #1">
        <p>If you can read this, tab #1 is expanded</p>
    </common-panel>
    <common-panel title="Panel #2">
        <p>If you can read this, tab #2 is expanded</p>
    </common-panel>
<common-panel-set>

The <common-panel-set> provides a DOM method like the <select> element, exposing .basic_panels collection, a basic_selectedIndex property, and a basic_selectedPanel property.

<common-panel-set>
    <common-panel id="bar" title="Panel #1">
        <p>If you can read this, tab #1 is expanded</p>
    </common-panel>
    <common-panel title="Panel #2" collapse-state="expanded">
        <p>If you can read this, tab #2 is expanded</p>
    </common-panel>
<common-panel-set>
<script>
    var panelSet = document.querySelector("#bar");
    console.log(panelSet.selectedIndex); // 1
    console.log(panelSet.basic_panels.length); // 2
    console.log(panelSet.basic_selectedPanel); // the selected basic panel element
<script>

Some examples of single select <common-panel-sets> are

  • tabset/tabmenu
  • accordion
  • carousel
  • deck
  • coverflow

A <common-panel> is defined as having:

  • An (optional) element for title (which allows use of CSS to ::before/::after) with attributes is-destroyable and collapse-state which can contain values "collapsed" or "expanded" (default)
  • A body element
